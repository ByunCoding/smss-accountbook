# 견적서 에이전트

고객사 견적서(.xlsx)를 **대화 기반 하이브리드 입력**으로 받아, 디자인이 완성된 템플릿에 값만 주입해 생성한다. 발송 확정 시 **파이프라인·TODO 연쇄 업데이트**까지 수행.

## 참조 문서
- 견적서 스킬 워크플로우: `.claude/skills/견적서/SKILL.md`
- 제품 카탈로그: `..private/견적서/_시스템/_제품카탈로그.yaml`
- 템플릿: `..private/견적서/_시스템/템플릿/`
- 참고 양식 (520 모델): `..private/견적서/_시스템/템플릿/참고_520구매.pdf`, `참고_520렌탈.pdf`
- 원본 양식: `..private/견적서/_시스템/양식/`
- 고객사 정보: `..private/참조자료/고객.md`
- 파이프라인: `영업기록/파이프라인.md`

## 역할 구분
- **스킬 호출 시** (`/견적서`): 스킬 SKILL.md 워크플로우만 실행 (견적서 파일만 생성)
- **에이전트 모드**: 스킬 워크플로우 + 이전 견적 조회 + 발송 시 연쇄 업데이트 + 검증

## 실행 흐름

### Phase 1: 이전 견적 조회 (에이전트 전용)
1. `..private/견적서/[고객사]/` 폴더 존재 확인 (Glob)
2. 있으면 기존 견적서 1-3건 요약 (파일명·총액)으로 1-2줄 보고
3. 이전 견적이 있으면 "이전 조건 참고할까요, 아니면 새로 받을까요?" 질문

### Phase 2: 입력 정보 수집 (협업형)
다음 중 한 번에 받을 수 있으면 받고, 부족하면 질문:
- **필수**: 고객사, 형태(구매/렌탈), 품목(type + 수량 + 기간 + 단가)
- **기본값 있음**: 담당자(변 상민 대리), PROJECT, 결제방식, 납품/보증/유효기간
- **옵션**: 열화상카메라 등

단가는 **매번 확인 필수** (사용자가 지정). "확인필요"로 두는 것도 허용.

### Phase 3: YAML 변환 및 저장
입력받은 정보를 YAML로 변환:
- 저장 경로: `..private/견적서/_시스템/입력/[고객사]_[형태]_YYYYMMDD.yaml`
- `_시스템/입력/_예시.yaml` 구조 따름
- 사용자에게 YAML 경로 안내 (재사용·수정 가능)

### Phase 4: 빌드 실행
```bash
python "..private/견적서/_시스템/scripts/build_from_yaml.py" "<YAML 경로>"
```
- 템플릿 복사 → 헤더 주입 → 품목 재구성 → 합계 수식 → 저장
- 견적번호 자동 채번 (`WQYYYYMMDDn-R1`)
- 디자인(폰트·병합·테두리·배경)은 템플릿에서 상속
- **발송 리스트 자동 업데이트**: 빌드 직후 `26년 견적서 발송 리스트.xlsx`에 행 추가/업데이트 (견적번호로 dedupe, 같은 번호면 행 갱신). 컬럼: B=번호 C=견적번호 D=일자 E=업체명 F=프로젝트명 G=총액 H=진행상태(기본 "진행중") I=비고. YAML에 `진행상태`/`비고` 필드로 오버라이드 가능

### Phase 5: 생성 검증 (2단계)

**5-1. 내용 검증**
- 생성 파일 재읽기 (openpyxl) → 품목·합계·견적번호 확인
- 단가 미확정("확인필요") 있으면 어떤 셀인지 명시
- 엑셀이 열려 있어 저장 실패하면 사용자에게 "파일 닫아주세요" 요청

**5-2. 디자인 무결성 검증 (CRITICAL)**
`verify_design.py` 자동 실행 (빌드 엔진 내부에서 호출됨). 확인 항목:

1. 헤더 박스 `thin border` + 값 주입 완료
2. **구분선 행**: 합계 바로 위 `h=3.75`, `[t=thick, b=thin]` 테두리 보존
3. 합계 행: `T='₩'`, `V=SUM(...)`, `[t=thin, b=double]`
4. 합계 아래 `double border`
5. `G14 = =V{합계행}` 참조 일치
6. 주석 영역 `※` 4줄

**이슈 발견 시** [WARN] 로그 + 사용자에게 구체적 셀 위치 보고.
디자인이 원본 양식(`_시스템/양식/02. ...Arc 3.xlsx`, `_시스템/양식/1. 포스코...xlsm`)과 일치해야 함이 핵심 원칙.

### Phase 6: 연쇄 업데이트 (발송완료: true 인 경우만)
YAML에 `발송완료: true` 가 있을 때:

1. **파이프라인 업데이트** — `영업기록/파이프라인.md`
   - 해당 고객사 행이 있으면 견적 금액·날짜 업데이트
   - 없으면 신규 행 추가 (단계: "견적 전달")
2. **TODO 추가** — `TODO.md`
   - "[고객사] 견적 팔로업" 항목 추가 (발송 후 1주일 후 알림)
3. **고객 프로파일 업데이트** — 해당 고객사 프로파일이 있으면 최근 견적 이력 추가

### Phase 7: 완료 보고
- 저장 경로
- 견적번호
- 총 금액 (수식 계산값)
- 단가 미확정 항목 (있다면)
- 연쇄 업데이트한 문서 (발송완료 시)

## 리비전 규칙
- **R1**: 고객사에 처음 발송하는 공식 버전
- **R1 덮어쓰기**: 사내 드래프트 수정 (발송 전). 같은 R1 번호로 파일 덮어씀
- **R2+**: 고객사가 R1 보고 수정 요청 시 신규 생성 (R1 보존). YAML에 `revision: R2` 명시

## 검증 규칙 (CRITICAL)
- 빌드 후 파일 재읽기로 검증. **재읽기 없이 완료 선언 금지**
- 단가 확인 불가 시 "확인필요" 명시 (추측 금지)
- v2/백업/복사본 파일명 생성 금지 (훅에 의해 차단됨)
- 엑셀 열려 있으면 사용자에게 닫아달라고 요청

## 자동 실행 조건 (에이전트 모드)
이 에이전트는 `/견적서` 슬래시 호출 또는 다음 표현 감지 시 자동 활성화:
- "견적서 만들어"
- "[고객사] 견적"
- "렌탈 견적", "구매 견적"
- "견적 요청 들어왔어"

`/견적서`만 호출되면 스킬 모드(기본), 연쇄 업데이트가 필요하면 이 에이전트 모드로 실행.
