# 퇴근정리

## 실행 순서

### Step 0: 문서점검 선행 (권장)
퇴근정리 본작업 **전에** `/문서점검` 스킬을 먼저 돌린다.
- 목적: 오늘 생긴 고객·TODO·파이프라인 변화가 일관되게 기록됐는지 먼저 확인 → 그 상태에서 일일기록·TODO 업데이트해야 왜곡 없음.
- 불일치 0건 → Step 1로 바로 진행
- 불일치 있으면 → 간단히 보고하고 "퇴근 전에 정리할까요, 내일로 넘길까요?" 확인
- 사용자가 "내일로"면 Step 1 진행 + 일일기록 **관찰·소감**에 "내일 문서점검 필요" 메모

### Step 1: 오늘 작업 파악
이번 세션 대화 내용을 바탕으로 오늘 완료한 것들을 정리한다.
- 저장된 파일 목록 확인 (미팅기록, CS케이스, 제안서 등)
- 완료된 항목, 진행 중인 항목, 미처리 항목 구분

### Step 2: 일일기록 생성/업데이트
`일일기록/YYYY-MM-DD.md` 파일을 오늘 날짜로 생성하거나 업데이트한다.
**전체 템플릿은 `일일기록/CLAUDE.md` 참조** — 아래는 핵심 골격.

```markdown
# YYYY-MM-DD (요일)

## 어제 MIT 가설 검증
- **어제 가설**: ...
- **실제 결과**: ...
- **차이·해석**: ...

## 오늘 한 일
-

## 관찰·소감
-

## 조용민 체크
-

## 내일 이어서 할 것
- ⭐ MIT: ...
  - 어떻게: ...
  - 기대 성과: ...
```

### Step 2-3: 어제 MIT 가설 검증 (피드백 분석 루프)
**어제 일일기록을 먼저 읽는다**. 어제 "내일 이어서 할 것"의 ⭐MIT 항목에 적힌 "어떻게/기대 성과"를 찾아낸다.
- 오늘 일일기록 맨 위 **어제 MIT 가설 검증** 섹션에 다음을 채운다:
  - **어제 가설**: 어제 적은 "어떻게 + 기대 성과" 그대로 인용
  - **실제 결과**: 오늘 실제 일어난 일
  - **차이·해석**: 예상대로 됐는지 / 빗나갔다면 왜 / 내가 잘 통하는 환경·방식 무엇인지
- 어제 가설이 없거나(이전 양식) MIT 자체가 없었으면 → "어제 가설 미기록 (이전 양식)" 한 줄 적고 다음 단계로.
- **목적**: 이형 "차별화 포인트" — 강점은 사전 가설 + 사후 검증 누적에서 발견됨. 이 칸을 빼면 일일기록은 사건 일지에 머무름.

### Step 2-5: 내일 MIT 1개 + 가설 확정
내일 할 일 중 가장 중요한 단 하나(MIT: Most Important Task)를 골라
일일기록 **내일 이어서 할 것** 섹션 맨 위에 다음 형식으로 표시한다.

```
- ⭐ MIT: [내일 가장 중요한 단 하나]
  - 어떻게: [어떤 방법·접근으로 풀 것인가 — 한 줄]
  - 기대 성과: [성공이라면 어떤 결과가 나와야 하나 — 한 줄, 가능하면 측정 가능하게]
```

기준: 이것 하나만 해도 오늘이 성공이라고 느낄 수 있는 것.
**"어떻게/기대 성과"는 반드시 함께 적는다** — 다음날 가설 검증의 입력값이 됨.

### Step 3: TODO.md 업데이트
TODO.md를 열고 다음 기준으로 업데이트한다:
- **오늘 할 일** → 완료된 것 삭제, 못 한 것은 내일로 이동
- **내일/다음 할 일** → 오늘 새로 생긴 후속 업무 추가
- **진행 중** → 멈춰있는 건 현황 반영

### Step 3-5: 조용민 일일 체크 (3문항)
일일기록의 **관찰·소감** 섹션에 아래 3가지를 추가한다.
사용자가 따로 언급 안 했어도 반드시 포함한다.

```
## 조용민 체크
- 오늘 고객에게 스펙 설명부터 했나, 문제 정의(Discovery)부터 했나?
- 오늘 "한 스텝 더(원마일)" 한 게 있었나? (아니면 왜 못 했나?)
- 오늘 역질문을 썼나, 아니면 고객/상사 말에 그냥 대답했나?
```

참조: `스크립트/언바운드_조용민/📋_분석요약.md`

### Step 3-7: 대시보드 & 포트폴리오 업데이트

오늘 기록을 바탕으로 두 HTML 파일을 업데이트한다.
**반드시 아래 확인 질문을 먼저 한 뒤** 답변을 기반으로 수치를 반영한다.

#### A. 업데이트 전 필수 확인 질문

오늘 아래 항목이 있었으면 **각각 질문한다** (해당 없는 항목은 생략):

**CS 처리 관련 활동이 있었으면:**
> "CS 건 — 네가 직접 처음부터 끝까지 해결한 거야, 아니면 부장님/다른 분이 주도하고 보조한 거야?"
- 직접 독립 처리 → **CS 처리 누계 +1**
- 보조/서포트 → 카운트 안 함, Activity에만 기록

**미팅/고객 접점이 있었으면:**
> "오늘 미팅 — 네가 기술 설명이나 제안을 직접 담당했어, 아니면 동행만 한 거야?"
- 직접 담당(기술 파트, 제안, 협의 주도) → **미팅 주도 누계 +1**
- 동행/관찰 → **미팅 동행(학습) 누계 +1** (별도 카운트)

**교육 진행이 있었으면:**
> "교육 — 네가 직접 앞에서 진행한 거야, 아니면 옆에서 지켜본 거야?"
- 직접 진행 → **교육 누계 +1**
- 참관 → 카운트 안 함

**고객사 담당 변화가 있었으면:**
> "새로 네가 담당하게 된 고객사 생겼어?"

#### B. KPI 분류 기준 (대시보드/포트폴리오 반영값)

| 항목 | 카운트 O (포트폴리오 어필용) | 카운트 X (기록만) |
|------|--------------------------|-----------------|
| CS 처리 | 내가 독립 처리한 건 | 보조, 팔로업만 한 건 |
| 미팅 | 내가 기술/제안 주도한 건 | 동행·관찰만 한 건 |
| 교육 | 내가 직접 진행한 건 | 참관한 건 |

대시보드에는 두 가지 모두 보여준다:
```
미팅 주도: N회   ← portfolio.html KPI에 반영
미팅 동행: N회   ← career_dashboard.html에만 표시 (성장 트래킹)
```

#### C. HTML 업데이트

확인 질문 답변 기반으로:
- **career_dashboard.html**: KPI 수치 + Recent Activity 추가 (5개 초과 시 오래된 것 제거)
- **portfolio.html**: "내가 직접 한 것" 수치만 반영

Recent Activity 도트 색상: CS 처리(독립) → `green`, 교육/기획 → `violet`, 미팅/외근 → `amber`

```html
<div class="activity-item">
  <div class="act-dot green"></div>
  <div class="act-body"><span class="act-title">CS 처리 완료</span> <span class="act-desc">· [고객사] [이슈 한 줄]</span></div>
  <span class="act-time">오늘</span>
</div>
```

오늘 특별히 기록할 활동이 없으면 Activity 추가만 하고 KPI는 유지.

### Step 4: 완료 보고
정리 결과를 한눈에 보여준다:
```
✅ 오늘 완료: N건
⭐ 내일 MIT: [한 줄]
📋 내일 TODO: N건
💾 일일기록 저장: 일일기록/YYYY-MM-DD.md
📊 대시보드 업데이트: [업데이트된 항목 한 줄 / 변동 없으면 "변동 없음"]
🚧 이번 주 블로커 (있으면): [상사에게 사전 공유가 필요한 장애물 — 없으면 생략]
```
